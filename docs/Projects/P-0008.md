## Scope 1: Caddy Infrastructure

**Branch:** `scope-P-0008-1-caddy-infra`
**Status:** `[ ]`
**Depends on:** none

### Tasks
- [ ] Install Caddy on Pi via official Debian repo (`apt install caddy`); verify `caddy.service` runs under systemd
- [ ] Add `PRD_DOMAIN` and `TST_DOMAIN` to `.env` config and document in `.env.example` (e.g., `PRD_DOMAIN=home.stanham.com`, `TST_DOMAIN=home.stanham.com`); `DEV_DOMAIN` defaults to `localhost` when absent
- [ ] Create `/etc/caddy/Caddyfile` that imports per-env include files from `/opt/latarnia/{env}/caddy/latarnia.caddyfile`
- [ ] Create `src/latarnia/caddy/manager.py` — `CaddyConfigManager` class with methods: `generate_config()`, `reload()`, `on_app_registered(app_id, port)`, `on_app_deregistered(app_id)`
- [ ] `generate_config()` reads `{ENV}_DOMAIN` from config and uses it as the Caddyfile site address; writes `/opt/latarnia/{env}/caddy/latarnia.caddyfile` with: public routes (`/auth/*`, `/docs`, `/openapi.json`), per-app swagger routes (`/apps/{app_name}/docs` and `/apps/{app_name}/openapi.json` — no `forward_auth`, no session required), per-app webUI routes with `forward_auth`, catch-all for dashboard/API/MCP. No `tls` directive — Caddy auto-provisions via ACME for real domains, self-signed for `localhost`. Per-app public routes are regenerated on every `on_app_registered` / `on_app_deregistered` call.
- [ ] `reload()` calls Caddy admin API `POST localhost:2019/load` with current config; log success/failure
- [ ] Add config write lock to `CaddyConfigManager` to prevent concurrent writes
- [ ] Wire `CaddyConfigManager` into `AppRegistry.register_app()` and `AppRegistry.deregister_app()` — call `generate_config()` then `reload()` after each
- [ ] Call `CaddyConfigManager.generate_config()` + `reload()` at Latarnia startup (after registry is loaded)
- [ ] Delete `src/latarnia/managers/web_proxy.py` and remove `WebProxyManager` import/mount from `main.py`
- [ ] Update `/api/apps/{id}/webui` proxy behavior in `main.py` — remove any remaining Python proxy calls; Caddy now handles it
- [ ] Add `ufw` rules to deployment runbook: `ufw allow 80/tcp` (ACME HTTP-01), `ufw allow 443/tcp`, `ufw allow 8443/tcp`, `ufw allow 22/tcp`, `ufw deny 8000/tcp`, `ufw deny 8100:8199/tcp`, `ufw deny 9001:9099/tcp`
- [ ] Create migration task: `docs/Projects/P-0008/migrations/001_ufw_setup.sh` — idempotent shell script applying the above ufw rules on the Pi (including port 80 for ACME)
- [ ] PRD listens on `{PRD_DOMAIN}:443`, TST on `{TST_DOMAIN}:8443` — verify generated Caddyfile site addresses match config
- [ ] Test: `curl https://{PRD_DOMAIN}` returns dashboard HTML (valid Let's Encrypt cert, no `-k` flag needed); direct `curl http://localhost:8000` from Pi loopback still works (internal); ufw blocks external test per acceptance criteria

### Acceptance Criteria
- **cap-001:** `systemctl is-active caddy` returns `active` on the Pi
- **cap-002:** Starting an app with `has_web_ui: true` triggers Caddyfile regeneration and reload; `https://latarnia.local/apps/{app_name}/` proxies to the app's port within 5 seconds of registration; deregistering the app causes reload and the route returns 404
- **cap-003:** Browser connects to `https://{PRD_DOMAIN}` with a valid Let's Encrypt certificate (no browser trust import required); HTTP on port 80 redirects to HTTPS; dev environment (`localhost`) uses Caddy's automatic self-signed cert
- **cap-004:** Unauthenticated `GET /` and `GET /apps/{app}/` redirect to `/auth/login`
- **cap-005:** `curl https://{PRD_DOMAIN}/apps/{app}/docs` returns 200 without session cookie; `curl https://{PRD_DOMAIN}/docs` returns 200 without session cookie
- **cap-006:** `curl http://<host-ip>:8000` from external host is refused (ufw DROP); `curl http://<host-ip>:8101` from external host is refused
- **cap-007:** `web_proxy.py` does not exist in the codebase; `WebProxyManager` is not imported anywhere

## Scope 2: Auth Foundation

**Branch:** `scope-P-0008-2-auth-foundation`
**Status:** `[ ]`
**Depends on:** Scope 1 (can be developed in parallel; integration testing requires Scope 1)

### Tasks
- [ ] Add `LATARNIA_TOTP_ENC_KEY` (32-byte base64 AES key) and `LATARNIA_JWT_SECRET` to `secrets.env` operator documentation; add both to `.env.example`
- [ ] Create `src/latarnia/auth/db.py` — `AuthDB` class: connects to `latarnia_platform_{env}`, creates DB if missing (using platform admin Postgres URL from config), runs migrations from `src/latarnia/auth/migrations/` in order; uses `schema_versions` table for idempotent execution
- [ ] Create auth migrations: `001_create_users.sql`, `002_create_sessions.sql`, `003_create_app_roles.sql`, `004_create_machine_tokens.sql` — per `data_model.md`
- [ ] Create `src/latarnia/auth/totp.py` — `generate_totp_secret()`, `encrypt_secret(plaintext, key)`, `decrypt_secret(ciphertext, key)`, `validate_code(encrypted_secret, code, key)` (AES-256-GCM; reject replayed codes via 30s window check)
- [ ] Create `src/latarnia/auth/sessions.py` — `create_session(user_id, ip) → token`, `validate_session(token) → user_id | None`, `invalidate_session(token)`, lazy cleanup of expired sessions
- [ ] Create `src/latarnia/auth/routes.py` — FastAPI router mounted at `/auth`:
  - `GET /auth/setup` — render TOTP setup page (redirect to `/auth/login` if user exists)
  - `POST /auth/setup` — validate first TOTP code, create user record, issue session
  - `GET /auth/login` — render login page
  - `POST /auth/login` — validate TOTP code, issue session cookie, redirect
  - `DELETE /auth/session` — logout (invalidate session)
  - `GET /auth/verify` — validate session, look up role for app in `X-Forwarded-Uri`, return `X-Latarnia-*` headers; return 200 or 401
- [ ] Create HTML templates for setup and login pages (Bootstrap 5, minimal, matches dashboard style)
- [ ] Mount `auth.routes` router in `main.py`; initialize `AuthDB` on startup
- [ ] Replay protection: store last-used TOTP code timestamp per user in `users` table; reject if same 30s window

### Acceptance Criteria
- **cap-008:** `latarnia_platform_{env}` DB exists after Latarnia startup; all 4 auth tables present; `schema_versions` tracks applied migrations
- **cap-009:** First visit to `/auth/setup` with no users renders QR code page; submitting valid TOTP code creates user and redirects; submitting invalid code returns error; second visit to `/auth/setup` redirects to `/auth/login`
- **cap-010:** Valid TOTP code → 302 redirect to `/` with `latarnia_session` cookie (HTTP-only, Secure, SameSite=Strict); invalid code → error, no cookie; replayed code → rejected
- **cap-011:** Session expires after configured TTL; expired session causes `/auth/verify` to return 401; `DELETE /auth/session` invalidates immediately
- **cap-012:** `GET /auth/verify` with valid session for `/apps/my_app/` returns 200 with `X-Latarnia-User` and `X-Latarnia-App-Role` headers; invalid/missing session returns 401; superuser session returns `X-Latarnia-Is-Super: true`
- **cap-013:** `users` table has one row after setup; TOTP secret column is encrypted ciphertext (not plaintext base32)

## Scope 4: JWT & Machine Tokens

**Branch:** `scope-P-0008-4-jwt-tokens`
**Status:** `[ ]`
**Depends on:** Scope 2, Scope 3

### Tasks
- [ ] Create `src/latarnia/auth/jwt_auth.py` — `issue_jwt(user_id, app_scope, expires_at) → str`, `validate_jwt(token) → claims | None`; HS256, signed with `LATARNIA_JWT_SECRET`; claims: `{sub, iat, exp, apps: {app_name: role}, super}`
- [ ] Add token management endpoints to `auth/routes.py`:
  - `POST /api/auth/tokens` — create JWT + store record (superuser required if any app has `full` role); return raw JWT once
  - `GET /api/auth/tokens` — list user's tokens (label, app_scope, expires_at, created_at, revoked_at)
  - `DELETE /api/auth/tokens/{id}` — revoke (set `revoked_at = now()`)
- [ ] Create `src/latarnia/auth/middleware.py` — `JWTAuthMiddleware`: on requests to `/api/*` or `/mcp/*`, check for `Authorization: Bearer <token>`; call `validate_jwt`; check `machine_tokens` table for revocation; attach claims to request state; return 401 if missing/invalid
- [ ] Apply `JWTAuthMiddleware` in `main.py` (or as a FastAPI dependency on API routes)
- [ ] Session-cookie auth (browser) and Bearer-token auth (machine) must coexist on the same API routes: if Bearer token is present, use JWT auth; if session cookie is present, use session auth; if neither, 401
- [ ] Update `mcp_gateway.py`: validate Bearer token before establishing MCP session; extract `apps` claim; only expose tools from apps in scope; inject `X-Latarnia-App-Role` header when connecting to per-app MCP server
- [ ] Implement "API Tokens" tab in dashboard UI (per wireframes.md): list tokens, new token form (label, app scope dropdowns, expiry), revoke button; show raw JWT once on creation (not stored)
- [ ] Create migration task: no new DB tables needed (machine_tokens created in Scope 2 migration 004)

### Acceptance Criteria
- **cap-019:** `POST /api/auth/tokens` returns signed JWT; token record in `GET /api/auth/tokens`; `DELETE /api/auth/tokens/{id}` marks revoked; subsequent API call with revoked token → 401
- **cap-020:** `GET /api/apps` with valid JWT → 200; no token → 401; expired JWT → 401; JWT scoped to app A used to access app B's endpoints → 403
- **cap-021:** MCP connection with valid JWT → tool list (scoped to JWT's app_scope); no JWT → 401; per-app MCP server receives `X-Latarnia-App-Role` header matching JWT claim

## Scope 5: Spec & Example App Update

**Branch:** `scope-P-0008-5-spec-examples`
**Status:** `[ ]`
**Depends on:** Scope 3, Scope 4

### Tasks
- [ ] Update `docs/System/app-specification.md` — add "Authentication Headers" section documenting:
  - `X-Latarnia-User: <user_id>` — always present on authenticated requests
  - `X-Latarnia-App-Role: <role>` — always present; values: `none | webUI-low | webUI-med | webUI-full | full`
  - `X-Latarnia-Is-Super: true | false`
  - Trust model: headers are set by Latarnia/Caddy; only trust them when app port is not directly reachable (firewall enforced)
  - Apps must not require auth themselves; they receive the user's role and may use it to adjust responses
  - Apps that do not implement role-based logic may ignore the headers entirely — backward compatible
- [ ] Update `docs/System/app-specification.md` — update "MCP Server Contract" section:
  - MCP connections receive `X-Latarnia-App-Role` header on the SSE connection from the gateway
  - Role level matches the user's webUI role for the app
  - MCP servers may optionally restrict tools based on this role
  - Add example: checking `X-Latarnia-App-Role` in a tool handler
- [ ] Update `examples/example_full_app/app.py`:
  - WebUI: read `X-Latarnia-App-Role` from request header; if `webUI-low`, hide write actions in the UI; if `full`, show admin section
  - MCP: in tool handlers, read role from a context variable set by the MCP server on connection (passed via header); restrict destructive tools to `full` and `webUI-full` roles
- [ ] Update `examples/example_full_app/latarnia.json` — no new fields needed; document in comments that the app uses role headers
- [ ] Verify `example_full_app` passes existing integration tests after changes

### Acceptance Criteria
- **cap-022:** `app-specification.md` contains "Authentication Headers" section with correct values table, trust model, and backward compatibility note; MCP section updated with role header documentation and example
