## Scope 2: Auth Foundation

**Branch:** `scope-P-0008-2-auth-foundation`
**Status:** `[DONE]`
**Depends on:** Scope 1 (can be developed in parallel; integration testing requires Scope 1)

> **Notes / decisions:** First (bootstrap) superuser username is `admin`
> (setup page collects only the code, per wireframe). Session cookie `Secure`
> flag is relaxed in `dev` only (plain http on :8000) so the browser keeps the
> cookie; prod/tst run behind Caddy TLS where `Secure` is always set. Current
> user is resolved from the validated session cookie (forwarded by Caddy),
> with the `X-Latarnia-User` header as a fallback. Setup/login redirect to
> `/dashboard` (the actual dashboard route; `/` returns JSON). `roles.py` lands
> in Scope 3 — until then `/auth/verify` resolves app role as `full` for
> superusers, `none` otherwise. Verified end-to-end via TestClient against real
> Postgres (`tests/integration/test_auth_flow.py`).

- [x] `LATARNIA_TOTP_ENC_KEY` + `LATARNIA_JWT_SECRET` documented in `.env.example` and runbook; loaded from `secrets.env`
- [x] `src/latarnia/auth/db.py` — `AuthDB`: creates `latarnia_platform_{env}` if missing, runs `migrations/` in order via `schema_versions` (idempotent)
- [x] Auth migrations 001–005 (users, user_credentials, sessions, app_roles, machine_tokens)
- [x] `src/latarnia/auth/providers/base.py` — `AuthProvider` Protocol
- [x] `src/latarnia/auth/providers/totp.py` — `TOTPAuthProvider` (AES-256-GCM, ±1 window, replay rejection via stored `last_totp_window`)

- [x] Mount auth router in `main.py`; `AuthDB.initialize()` on startup
- [x] Replay protection via `last_totp_window` in `credential_data`

### Acceptance Criteria
- **cap-008:** `latarnia_platform_{env}` DB exists after Latarnia startup; all 5 auth tables present (`users`, `user_credentials`, `sessions`, `app_roles`, `machine_tokens`); `schema_versions` tracks applied migrations
- **cap-009:** First visit to `/auth/setup` with no users renders QR code page; submitting valid TOTP code creates superuser and redirects; submitting invalid code returns error; `POST /api/auth/users` returns `{setup_url}`; visiting `/auth/setup?token=<token>` renders QR for new user; submitting valid code activates new user; setup token is single-use
- **cap-010:** Login page has username + code fields; valid username + TOTP code → 302 redirect to `/` with `latarnia_session` cookie (HTTP-only, Secure, SameSite=Strict); invalid code or unknown username → error, no cookie; replayed code → rejected

- **cap-013:** `user_credentials` row for the user has `auth_method = 'totp'`; `credential_data.totp_secret_enc` is encrypted ciphertext (not plaintext base32)
- **cap-024:** `GET /api/auth/users` returns user list; `POST /api/auth/users` (superuser) returns setup URL; `DELETE /api/auth/users/{id}` deactivates user and invalidates sessions; non-superuser requests → 403; superuser cannot deactivate themselves → 403

- [ ] Create `src/latarnia/auth/middleware.py` — `JWTAuthMiddleware`: on requests to `/api/*` or `/mcp/*`, check for `Authorization: Bearer <token>`; call `validate_jwt`; check `machine_tokens` table for revocation; attach claims to request state; return 401 if missing/invalid
- [ ] Apply `JWTAuthMiddleware` in `main.py` (or as a FastAPI dependency on API routes)
- [ ] Session-cookie auth (browser) and Bearer-token auth (machine) must coexist on the same API routes: if Bearer token is present, use JWT auth; if session cookie is present, use session auth; if neither, 401
- [ ] Update `mcp_gateway.py`: validate Bearer token before establishing MCP session; extract `apps` claim; only expose tools from apps in scope; inject `X-Latarnia-App-Role` header when connecting to per-app MCP server

- [ ] Create migration task: no new DB tables needed (machine_tokens created in Scope 2 migration 004)

### Acceptance Criteria
- **cap-019:** `POST /api/auth/tokens` returns signed JWT; token record in `GET /api/auth/tokens`; `DELETE /api/auth/tokens/{id}` marks revoked; subsequent API call with revoked token → 401
- **cap-020:** `GET /api/apps` with valid JWT → 200; no token → 401; expired JWT → 401; JWT scoped to app A used to access app B's endpoints → 403
- **cap-021:** MCP connection with valid JWT → tool list (scoped to JWT's app_scope); no JWT → 401; per-app MCP server receives `X-Latarnia-App-Role` header matching JWT claim

### Tasks
- [ ] Update `docs/System/app-specification.md` — add "Authentication Headers" section documenting:
  - `X-Latarnia-User: <user_id>` — always present on authenticated requests
  - `X-Latarnia-App-Role: <role>` — always present; values: `none | webUI-low | webUI-med | webUI-full | full`
  - `X-Latarnia-Is-Super: true | false`
  - Trust model: headers are set by Latarnia/Caddy; only trust them when app port is not directly reachable (firewall enforced)
  - Apps must not require auth themselves; they receive the user's role and may use it to adjust responses
  - Apps that do not implement role-based logic may ignore the headers entirely — backward compatible
- [ ] Update `docs/System/app-specification.md` — update "MCP Server Contract" section:
  - MCP connections receive `X-Latarnia-App-Role` header on the SSE connection from the gateway
  - Role level matches the user's webUI role for the app
  - MCP servers may optionally restrict tools based on this role
  - Add example: checking `X-Latarnia-App-Role` in a tool handler
- [ ] Update `examples/example_full_app/app.py`:
  - WebUI: read `X-Latarnia-App-Role` from request header; if `webUI-low`, hide write actions in the UI; if `full`, show admin section

- [ ] Update `examples/example_full_app/latarnia.json` — no new fields needed; document in comments that the app uses role headers
