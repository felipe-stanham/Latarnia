## Scope 1: Routing & redirects

**Branch:** `scope-P-0010-1-routing-redirects`
**Status:** `[ ]` → `[IN PROGRESS]` → `[DONE]`
**Depends on:** none

### Tasks
- [ ] In `main.py` (~line 449), replace the existing `GET /` JSON probe
      (`{"message": "Latarnia is running", ...}`) with
      `RedirectResponse("/dashboard", 302)`. `/health` remains the liveness
      probe. Grep tests/scripts for assertions on the old JSON body and update
      (none found at spec time).
- [ ] In `caddy/manager.py` `_forward_auth_lines()` (~line 115), change the
      `forward_auth` redirect from `next={http.request.uri.path}` to
      `next={http.request.orig_uri}` (one emit site covers both the catch-all
      and per-App blocks).
- [ ] Update the two `caddy_manager` unit tests asserting the redirect line
      (`tests/unit/test_caddy_manager.py` ~lines 115, 125).
- [ ] Validate the generated tst Caddyfile with `caddy validate` on the Pi.
- [ ] In `auth/routes.py`, add a `safe_next(next: str|None) -> str` helper and
      use it in `post_login` (~line 241, replacing the bare
      `next.startswith("/")` check): accept `next` only if it starts with a
      single `/`, does not start with `//` or `/\`, and contains no `://`;
      otherwise return `/dashboard`. Must tolerate `?` in otherwise-safe values
      (`orig_uri` includes the query string — see spec.md "Caddy orig_uri" for
      the accepted encoding limitation).

### Acceptance Criteria
> Copied verbatim from spec.md.

- **root_redirects_authenticated:** authenticated `GET /` (valid session cookie) → `302` with `Location: /dashboard`.
- **root_unauth_still_login:** unauthenticated `GET /` (no cookie, browser `Accept`) → `302 → /auth/login` (Caddy forward_auth, unchanged).
- **login_returns_to_app_path:** unauthenticated browser GET `/apps/latarnik/` → redirected to `/auth/login?next=/apps/latarnik/`; after valid `POST /auth/login` with that `next` → `303` with `Location: /apps/latarnik/`.
- **login_default_dashboard:** valid `POST /auth/login` with no `next` → `303` with `Location: /dashboard`.
- **login_rejects_open_redirect:** `POST /auth/login` with `next=//evil.com` (or `next=https://evil.com` or `next=/\evil.com`) → login succeeds but redirect target falls back to `/dashboard` (never an off-site host).
- **caddy_emits_orig_uri:** generated Caddyfile `forward_auth` redirect uses `next={http.request.orig_uri}`; `caddy validate` accepts it.

## Scope 3: Activity feed filtering + WS gate

**Branch:** `scope-P-0010-3-activity-filter`
**Status:** `[ ]` → `[IN PROGRESS]` → `[DONE]`
**Depends on:** Scope 2 (uses the shared helper / current-user resolution)

> Source model (verified, see spec.md "Activity source model"): an event's
> `source` is publisher-set in the XADD (convention: the manifest App name,
> e.g. `example_full_app`), falling back to the stream-name suffix when absent.
> There is no `app_id` in events and no platform publisher emitting
> `source=system` — do NOT build an app_id→name mapping or enumerate system
> sources. Filter is default-deny on App name.

### Tasks
- [ ] In `GET /api/activity/recent` (~main.py:1071), resolve the current user
      (session or Bearer). Superuser → return all events unchanged.
- [ ] Non-Superuser → keep an event **iff** its `source` equals the name of a
      registered App (App Registry) **and** `get_role(user_id, app_name) ==
      "full"` (`auth/roles.py`). Drop everything else — system strings, raw
      stream names, unknown sources (default-deny).
- [ ] Gate the WebSocket `/ws/activity` (~main.py:1135) Superuser-only: resolve
      the session user from the cookie at connect; if not Superuser, close
      (e.g. code 4403) before any event is sent.
- [ ] In `templates/dashboard.html`, open the activity WebSocket
      (`connectActivityWebSocket()`, ~line 1120/1376) only when the page is
      rendered for a Superuser (reuse the Scope 2 template flag); non-Superusers
      get the filtered REST list on refresh (manual-refresh pattern).

### Acceptance Criteria
> Copied verbatim from spec.md.

- **activity_filtered_for_user:** given recent events with sources `latarnik` (user role = `full`) and `other_app` (user role = `none`), `GET /api/activity/recent` as that user returns only `latarnik` events.
- **activity_default_deny_for_user:** an event whose `source` does not resolve to a registered App name (e.g. `system`, a raw stream-name suffix like `example.events.created`, or an unknown string) is absent from a non-Superuser's `GET /api/activity/recent`.
- **activity_full_for_super:** `GET /api/activity/recent` as a Superuser returns all events including unresolvable sources and apps the Superuser has no explicit role on.
- **activity_ws_superuser_only:** a non-Superuser's WebSocket connect to `/ws/activity` is refused/closed before any event is delivered; a Superuser's connect succeeds and receives events.

## Scope 4: User management

**Branch:** `scope-P-0010-4-user-management`
**Status:** `[ ]` → `[IN PROGRESS]` → `[DONE]`
**Depends on:** none (schema scope — owns migration 006)

> ⚠️ Route migration: `DELETE /api/auth/users/{id}` currently means
> **deactivate** (`auth/routes.py` ~line 315). In ONE commit: move deactivation
> to `POST /api/auth/users/{id}/deactivate`, repurpose `DELETE` as hard-delete,
> and update both callers — `deactivateUser()` in `templates/dashboard.html`
> (~line 1252) and `tests/integration/test_auth_flow.py` (~lines 201–210).

### Tasks
- [ ] Add migration `006_granted_by_set_null.sql` (drop + re-add the two
      `granted_by` FKs as `ON DELETE SET NULL`). The FKs are unnamed in
      migrations 004/005, so the Postgres default names
      (`app_roles_granted_by_fkey`, `machine_tokens_granted_by_fkey`) apply —
      confirm on the Pi (`\d app_roles`, `\d machine_tokens`) before finalizing.
- [ ] `auth/tokens.py`: `revoke_all_for_user(user_id)` — `UPDATE machine_tokens
      SET revoked_at = NOW() WHERE user_id = %s AND revoked_at IS NULL`. No
      middleware change (it already rejects revoked/missing rows via
      `is_active`).
- [ ] `auth/users.py`: `delete_user(user_id, requester_id)` with guards (no
      self-delete; no deleting the last `is_active AND is_superuser` user);
      relies on cascade (children, incl. `machine_tokens` → JWTs die) + SET NULL
      (grant attributions).
- [ ] `auth/users.py`: extend `deactivate_user` to also call
      `revoke_all_for_user` (it already deletes sessions).
- [ ] `auth/users.py`: `reactivate_user(user_id)` — set `is_active = TRUE`;
      reject (409) if the user has no TOTP credential
      (`totp_provider.get_existing_secret` is the existing check). Does NOT
      un-revoke machine tokens.
- [ ] `auth/users.py` + `providers/totp.py`: `reissue_setup_token(user_id)` —
      delete the user's `totp` credential, delete their sessions, revoke their
      machine tokens, set `is_active = FALSE`, mint a new `setup_token`
      (+24h expiry, same `secrets.token_urlsafe(32)` pattern as `create_user`);
      return the `setup_url`. (Deleting the credential is what makes
      `/auth/setup`'s `ensure_credentials` mint a FRESH secret — verified.)
- [ ] `auth/routes.py`: Superuser-gated endpoints —
      `DELETE /api/auth/users/{id}` (hard-delete),
      `POST /api/auth/users/{id}/deactivate` (moved),
      `POST /api/auth/users/{id}/activate`,
      `POST /api/auth/users/{id}/setup-token`.
- [ ] `templates/dashboard.html`: repoint `deactivateUser()` to the new route;
      add Delete / Activate / Re-issue setup controls + confirm dialogs in
      Users & Roles (per the wireframes — confirm copy mentions API-token
      revocation); show the returned setup URL with a Copy button.
- [ ] `tests/integration/test_auth_flow.py`: move deactivation assertions to the
      new route; add delete / reactivate / re-issue coverage per the criteria.
- [ ] Verify re-enrollment via existing `/auth/setup?token=…` mints a fresh
      secret and re-activates the user.
- [ ] Update `docs/System/dataModel.md` to record the `ON DELETE SET NULL`
      change and the revocation semantics of `machine_tokens.revoked_at`.

### Acceptance Criteria
> Copied verbatim from spec.md.

- **delete_removes_user_and_children:** Superuser `DELETE /api/auth/users/{id}` on an active non-Superuser → `200`; the `users` row and its `user_credentials`/`sessions`/`app_roles`/`machine_tokens` rows are gone.
- **delete_rejects_machine_tokens:** a JWT issued to the deleted user is rejected (`401`) after the delete (its `machine_tokens` row is gone; `is_active` lookup fails).
- **delete_preserves_granted_rows:** if the deleted user had granted an `app_role` to another user, that role row survives with `granted_by = NULL`.
- **delete_self_forbidden:** Superuser `DELETE /api/auth/users/{own_id}` → `409` (or `403`); the row remains.
- **delete_last_superuser_forbidden:** `DELETE` of the only active Superuser → `409`; the row remains.
- **deactivate_moved_route:** `POST /api/auth/users/{id}/deactivate` deactivates (Superuser-only; `is_active = FALSE`, sessions deleted, machine tokens revoked); the old deactivate-via-`DELETE` semantics no longer exist. Dashboard Deactivate button and existing integration tests use the new route.
- **reactivate_user:** Superuser `POST /api/auth/users/{id}/activate` on a deactivated user that has a TOTP credential → `200`; `is_active` becomes `TRUE`; the user can log in with their existing authenticator.
- **reactivate_requires_credential:** `POST /api/auth/users/{id}/activate` on a user with no `user_credentials` row → `409` with a message to re-issue setup instead.
- **reactivate_keeps_tokens_revoked:** reactivation does not clear `revoked_at` on the user's machine tokens; a token revoked by deactivate/re-issue stays rejected.
- **reissue_returns_setup_url:** Superuser `POST /api/auth/users/{id}/setup-token` → `200` with a `setup_url` of the form `/auth/setup?token=<token>`; the user's `setup_token`/`setup_token_expires_at` are set, `is_active` becomes `FALSE`, their `sessions` are deleted, and their existing `totp` `user_credentials` row is removed.
- **reissue_revokes_machine_tokens:** after re-issue, all the user's machine tokens have `revoked_at` set; a previously-issued JWT for that user → `401`.
- **reissue_then_enroll_new_secret:** completing `POST /auth/setup?token=<token>` with a valid code generates a NEW TOTP secret (distinct credential row) and re-activates the user.
- **reissue_invalidates_old_device:** after re-issue and before re-enrollment, a code from the user's old authenticator fails login (credential rotated/removed).
