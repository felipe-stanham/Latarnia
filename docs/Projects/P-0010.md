## Scope 4: User management

**Branch:** `scope-P-0010-4-user-management`
**Status:** `[ ]` â†’ `[IN PROGRESS]` â†’ `[DONE]`
**Depends on:** none (schema scope â€” owns migration 006)

### Tasks

- [ ] `auth/users.py`: `reactivate_user(user_id)` â€” set `is_active = TRUE`; reject

- [ ] `auth/users.py` + `providers/totp.py`: `reissue_setup_token(user_id)` â€”
      delete the user's `totp` credential, delete their sessions, set
      `is_active = FALSE`, mint a new `setup_token` (+24h expiry); return the
      `setup_url`.
- [ ] `auth/routes.py`: Superuser-gated endpoints
      `DELETE /api/auth/users/{id}`, `POST /api/auth/users/{id}/activate`,
      `POST /api/auth/users/{id}/setup-token`.
- [ ] `dashboard.html`: add Delete / Activate / Re-issue setup controls + confirm
      dialogs in Users & Roles (per the wireframes); show the returned setup URL.
- [ ] Verify re-enrollment via existing `/auth/setup?token=â€¦` mints a fresh secret.
- [ ] Update `docs/System/dataModel.md` to record the `ON DELETE SET NULL` change.

### Acceptance Criteria
> Copied verbatim from spec.md.

- **delete_removes_user_and_children:** Superuser `DELETE /api/auth/users/{id}` on an active non-Superuser â†’ `200`; the `users` row and its `user_credentials`/`sessions`/`app_roles`/`machine_tokens` rows are gone.
- **delete_preserves_granted_rows:** if the deleted user had granted an `app_role` to another user, that role row survives with `granted_by = NULL`.
- **delete_self_forbidden:** Superuser `DELETE /api/auth/users/{own_id}` â†’ `409` (or `403`); the row remains.
- **delete_last_superuser_forbidden:** `DELETE` of the only active Superuser â†’ `409`; the row remains.
- **reactivate_user:** Superuser `POST /api/auth/users/{id}/activate` on a deactivated user that has a TOTP credential â†’ `200`; `is_active` becomes `TRUE`; the user can log in with their existing authenticator.
- **reactivate_requires_credential:** `POST /api/auth/users/{id}/activate` on a user with no `user_credentials` row â†’ `409` with a message to re-issue setup instead.
- **reissue_returns_setup_url:** Superuser `POST /api/auth/users/{id}/setup-token` â†’ `200` with a `setup_url` of the form `/auth/setup?token=<token>`; the user's `setup_token`/`setup_token_expires_at` are set, `is_active` becomes `FALSE`, their `sessions` are deleted, and their existing `totp` `user_credentials` row is removed.
- **reissue_then_enroll_new_secret:** completing `POST /auth/setup?token=<token>` with a valid code generates a NEW TOTP secret (distinct credential row) and re-activates the user.
- **reissue_invalidates_old_device:** after re-issue and before re-enrollment, a code from the user's old authenticator fails login (credential rotated/removed).
