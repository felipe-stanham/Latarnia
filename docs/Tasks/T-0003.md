**Relates to:** P-0008 (Scope 1 Caddyfile generation)

## Problem

`CaddyConfigManager.generate_config()` emits `forward_auth` blocks (the catch-all
`handle /*` and per-app `handle_path /apps/{name}/*`) with only `uri` +
`copy_headers`. When `/auth/verify` returns **401** (no/expired session), Caddy
relays that bare 401 to the client. An unauthenticated browser therefore lands
on a blank `401` page instead of being redirected to the login screen.

This violates the spec:
- `P-0008/spec.md:79` — "Unauthenticated `GET /` redirects to `/auth/login`"
- `P-0008/spec.md:80` — "Unauthenticated `GET /apps/{app}/` redirects to `/auth/login`"
- `P-0008/workflows.md:85-86` — `AuthVerify → Caddy: 401`, then `Caddy → Browser: 302 → /auth/login`

Discovered during the P-0008 tst rollout (Phase C verification): security is
intact (unauthenticated access is blocked, `/api/*` correctly requires a JWT),
but the browser redirect to login is missing.

## Fix

In `generate_config()`, both `forward_auth` blocks gain a response handler that
turns a 401 into a `302 → /auth/login`, preserving the original path as `next`
(the login route already honors `?next=/path`):

```
forward_auth localhost:8000 {
    uri /auth/verify
    copy_headers X-Latarnia-User X-Latarnia-App-Role X-Latarnia-Is-Super
    @auth_denied status 401
    handle_response @auth_denied {
        redir * /auth/login?next={http.request.uri.path} 302
    }
}
```

Only **401** is matched (not 403) — a logged-in user lacking an app role gets a
403 from the app, which must not redirect-loop to login. The `/api/*` and
`/mcp/*` blocks stay untouched (no `forward_auth`; machine clients get a real
401, never a redirect).

Factored into a `_forward_auth_lines()` helper since both call sites need the
identical block.
