# T-0002: Carve `/api/*` and `/mcp/*` out of Caddy `forward_auth`

**Created:** 2026-06-02 | **Status:** [DONE] | **Branch:** `task-T-0002-caddy-api-mcp-bypass`

> **Done:** `generate_config()` now emits `handle /api/*` and (when MCP enabled)
> `handle {gateway_path}/*` blocks with no `forward_auth`, before the catch-all.
> Unit tests added (t2-001..003); full suite green (396 passed). Spec
> `P-0008/architecture.md` Caddyfile example + note and `docs/System`
> architecture/workflows updated to match.
**Relates to:** P-0008 (Scope 1 Caddyfile generation + Scope 4 JWT)

## Problem

`CaddyConfigManager.generate_config()` routes `/api/*` and `/mcp/*` under the
catch-all `handle /*` block, which uses `forward_auth` → `/auth/verify`.
`/auth/verify` only validates the **session cookie**. A machine client sending
an `Authorization: Bearer <jwt>` to `/api/*` or `/mcp/*` *through Caddy* is
therefore rejected by `forward_auth` before reaching Latarnia's
`JWTAuthMiddleware` / MCP gateway, which are the components that actually
validate Bearer tokens. Net effect: machine (JWT) access works against Latarnia
directly (`:8000`) but not end-to-end through Caddy on tst/prd.

The intended design (per `P-0008/architecture.md`: "machine clients bypass
Caddy's forward_auth …") is that `/api/*` and `/mcp/*` are proxied straight to
Latarnia, which authenticates Bearer-or-session internally.

## Fix

In `generate_config()`, emit dedicated blocks **before** the catch-all:
- `handle /api/* { reverse_proxy localhost:{main_port} }` — no `forward_auth`
  (Latarnia's `JWTAuthMiddleware` enforces Bearer JWT or session cookie).
- `handle {gateway_path}/* { reverse_proxy localhost:{main_port} }` — no
  `forward_auth` (the MCP gateway validates the Bearer JWT internally). Emitted
  only when `config.mcp.enabled`.

The dashboard/webUI catch-all (`handle /*`) keeps `forward_auth`. Per-app webUI
blocks (`/apps/{name}/*`) keep `forward_auth` (browser session; the app trusts
the injected `X-Latarnia-App-Role`).

## Acceptance Criteria

- **t2-001:** Generated Caddyfile contains a `handle /api/* {` block whose body
  has **no** `forward_auth` and reverse-proxies to the platform port.
- **t2-002:** When MCP is enabled, generated Caddyfile contains a
  `handle /mcp/* {` block with **no** `forward_auth`; absent when MCP disabled.
- **t2-003:** The catch-all `handle /* {` block still contains `forward_auth`
  with `uri /auth/verify`, and per-app `/apps/{name}/*` blocks still use
  `forward_auth`.
- **t2-004:** Full unit suite green.
